TITLE  : Command Handlers: dispatcher arms and per-domain effects
OVERLAP: Complements docs/command-pipeline.md (the obvious sibling) — that doc walks one command end-to-end through client/parser/RoomEngine/dispatcher/queue and lists dispatcher 'big moves' + pitfalls at a flow level, but explicitly stops at 'ApplyCommand is a thin routing switch → handlers' and never documents the two switch surfaces, the handler-authoring contract, or per-domain effects. Net new content: the ApplyCommand-vs-TryApplyTowerCommand split, the handler read/write contract, and the per-domain effect cheat-sheet — none of which exists anywhere today.
EXPLORE COST: ~12,000 LOC cold across 9 files in src/Yaat.Sim/Commands/ (CommandDispatcher 2,468; ApproachCommandHandler 1,849; GroundCommandHandler 1,535; NavigationCommandHandler 1,489; DepartureClearanceHandler 1,462 — wait, PatternCommandHandler 2,028; FlightCommandHandler 931; ContactCommandHandler 189; FlightPlanCommandHandler 41; DispatchContext 46). To understand it cold an agent must hold BOTH dispatcher switch surfaces in head (ApplyCommand ~360 lines, TryApplyTowerCommand ~270 lines) and chase ~8 cross-references out of Commands/: Phase.CanAcceptCommand / CommandAcceptance (phases.md, Phases/), ControlTargets fields written by handlers (Simulation/ControlTargets.cs), PhaseList install/Clear semantics, CommandDescriber.GetCommandDimension + CommandDimension flags, CommandQueue/CommandBlock/BlockTrigger, ToSnapshot/FromSnapshot for the dry-run clone, and CommandResult/RejectedCommandType. Realistically a half-day read for someone who hasn't seen it.
WHY IT HELPS: Every command feature/fix edits a handler, and the dispatcher is the highest-churn, largest source subsystem (~12K LOC). Today an agent implementing 'make verb X do Y' must reverse-engineer two undocumented questions each time: (1) which of the two switch surfaces does my verb belong in — ApplyCommand or TryApplyTowerCommand — and why some verbs are duplicated in both; (2) what is the handler actually allowed/expected to mutate (ControlTargets fields vs installing a PhaseList) and how does it signal accept/reject/clears-phase to the phase gate. The existing command-pipeline.md deliberately stops at 'ApplyCommand is a thin routing switch over command type → handlers' and never opens the handler box. The architecture.md task index gives only a one-line file list ('CommandRegistry → CommandScheme → CommandSchemeParser → CommandDispatcher → handler'). Neither explains the contract. This doc amortizes the half-day cold read into a 15-minute reference and prevents the recurring class of bugs the code comments already memorialize: a queued tower verb hitting the no-arm fallback, phases torn down on an invalid command, SAY firing twice in dry-run, a handler calling Queue.Clear() instead of relying on dimension-aware clearing, or a new verb peeling an IFR departure off runway heading.

FOOTGUNS:
   ! Two switch surfaces, not one. ApplyCommand handles airborne/nav/flight/squawk/say; TryApplyTowerCommand handles phase-interactive tower & ground verbs (CTO, LUAW, CLAND, pattern turns, TAXI, CROSS, hold-short). Several verbs (CLAND, GoAround, pattern entries, holds) appear in BOTH switches because they can arrive with or without an active phase. Add a phase-interactive verb to only ApplyCommand and a queued/triggered instance will hit the __NO_DISPATCHER_ARM__ fallback when it re-fires after a phase transition.
   ! __NO_DISPATCHER_ARM__ and PhaseShouldBeCleared are string/identity sentinels, not exceptions. The default arm of ApplyCommand returns a CommandResult whose message starts with the marker; DryRunApplyCommand and BuildApplyAction detect it by substring/ReferenceEquals. Returning a generic failure instead silently breaks the tower-fallback routing.
   ! Phase clearing is deferred until after dry-run validation. DispatchWithPhase returns the PhaseShouldBeCleared sentinel rather than clearing in place — clearing before validation would destroy pattern/approach state on a command that then fails. The same dance is re-implemented inside BuildApplyAction for triggered blocks; both sites must stay in sync.
   ! Dry-run runs the first block on a clone (AircraftState.FromSnapshot). Handlers must be clone-safe: any write to a singleton, a sibling aircraft, or anything not on the cloned AircraftState leaks out of the dry run. TerminalEmitter is nulled in the dry-run context specifically so SAY-class verbs don't broadcast phantom pilot transmissions.
   ! Handlers must never call aircraft.Queue.Clear() or aircraft.CommandQueue.Clear() to 'reset' state. Queue clearing is dimension-aware (ClearConflictingBlocks + SplitBlockNonConflicting): a heading command clears queued lateral blocks but preserves a queued altitude block. A handler that wipes the whole queue defeats parallel-block survival.
   ! Handlers write ControlTargets and never touch position/heading directly — EXCEPT Force* verbs (ApplyForceHeading/ApplyForceAltitude/WARP), which deliberately teleport by writing aircraft.TrueHeading/Altitude/Position. Those are sim-control bypasses (IsSimControlBypass) that skip the phase gate because they wipe phase/queue/route inside the handler.
   ! VFR/IFR gating lives in the dispatcher, not the handlers: RequiresVfr() rejects pattern/option verbs for IFR aircraft, and CheckIfrDepartureCompatibility() rejects pattern-relative CTO modifiers for IFR departures so they don't peel off runway heading at liftoff. A new pattern-ish verb must be added to RequiresVfr or it will be accepted for IFR traffic.
   ! Transparent compounds (squawk, ident, say, RFIS/RTIS, NODEL, wake) bypass phases AND queue clearing via the IsAllTransparent fast path — and this fires whether or not a phase is active, because the same None-dimension path that wipes the queue under an active phase also wipes it when phases are null. A 'harmless' status verb misclassified as non-transparent can wipe a queued pattern entry.
   ! Installing a phase = build a fresh PhaseList, Clear() the old one with a PhaseContext, Add phases, then Start() with another PhaseContext (see DispatchJfac / TryAirborneFollow / DispatchHoldingPattern). Skipping the Clear()/Start() lifecycle leaves stale phase indices or unstarted phases. Use CommandDispatcher.BuildMinimalContext to construct the PhaseContext.
   ! NotifyPhaseCommandAccepted releases phase-internal holds (e.g. RV-SID runway-heading hold) only AFTER a successful apply, both on immediate dispatch and at trigger-fire time in BuildApplyAction. The Unsupported/transparent/sim-control guards inside it are load-bearing for the queued-block path even though they look redundant for immediate dispatch — the comment in code warns against removing them.

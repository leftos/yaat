# M11.1 — Variable pilot competence per aircraft

**Status:** Stub — not yet planned. Flesh out before implementing.

## Why

Today every simulated pilot in solo mode reads back perfectly, complies precisely, and reacts in the same fixed beat. Real pilots vary: a 200-hr student in a 172 lags on readbacks and drifts altitude; an ATP in a 737 nails everything. Variability is what forces the controller to actually listen to readbacks and re-issue when needed — without it the radio half of the simulation degrades into a deterministic event log.

## Sketch

- Per-spawn `PilotProficiency` enum (`Student`, `Normal`, `Sharp`) seeded from operator class + a small random draw. Stored on `AircraftState`.
- Affects three knobs:
  - **Readback completeness** — `Student` drops more elements in busy radio conditions (M10.3.5 already shortens by activity; this multiplies the shortening for `Student` and disables it for `Sharp`).
  - **Reaction latency** — slug between command dispatch and visible compliance (turn rate ramp-up, climb initiation). Tiny absolute window (~1–3 s) so it doesn't break tests.
  - **Tracking precision** — `Student` slight altitude/heading overshoot before settling.
- Plumbs through `PilotResponder` for readbacks; through `FlightPhysics` for compliance latency / precision.
- Scenario authors can pin a specific aircraft's proficiency for deliberate drills.

## Open questions

- Does proficiency persist across handoffs, or is it a property of "the pilot we hear on frequency"? (Likely persistent — same crew flies the leg.)
- Should the controller see proficiency anywhere in the UI? (Probably no — the whole point is they figure it out from behavior.)
- How do we keep tests deterministic? (Default to `Normal` in tests; opt-in for proficiency-aware fixtures.)
- Where's the cutoff between "realistic imperfection" and "broken sim"? Need acceptance criteria so a `Student` doesn't fail to comply badly enough to look like a bug.

## Out of scope (probably)

- Modelling pilot fatigue, distraction, or "captain takes over" hand-offs (M11.9 covers some of this).
- Foreign accent or audio quality variation (out of scope for solo, would need new TTS work).

## Previously deferred

The README lists **"Response timing jitter"** as deferred indefinitely with rationale "deterministic readbacks are easier to test and learn against." This milestone reopens that decision — the call now is that the training value (forcing the controller to listen) outweighs the test-determinism cost. The fix is to pin proficiency to `Normal` in tests, not to skip the feature.

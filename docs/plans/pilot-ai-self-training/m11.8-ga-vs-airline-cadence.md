# M11.8 — GA vs airline pilot cadence

**Status:** Stub — not yet planned. Confirm scope with the user before flesh-out — this re-opens a deferred decision.

## Why

GA pilots and airline pilots sound different on the radio: airline crews are precise, fast, and use standard phraseology; GA pilots are often slower, more verbose, less formal. Same readback content, very different feel. With one uniform voice the simulator collapses this distinction and the controller misses the cue ("oh, this is a Cessna pilot — they might need more time").

## Previously deferred

The README lists **"Richer pilot personality / callsign abbreviation / frequency compression"** as deferred indefinitely with rationale:
> M10.3.5 ships activity-aware shortcuts for safe command families plus light quiet-frequency flavor. Broader style changes need more aircraft/frequency context to avoid ambiguous or nonstandard readbacks.

Reopening this is a step beyond M10.3.5 — instead of activity-aware shortcuts, we want **operator-class-aware** style. The risk noted in the deferral ("ambiguous or nonstandard readbacks") still applies and must be respected: any style variation has to remain unambiguous.

## Sketch

- Use existing operator class (airline vs GA vs military) already encoded in the callsign / scenario.
- Per-class style profile affecting:
  - **Phraseology compression** — airline crews drop pleasantries ("good day"), GA includes them.
  - **Speech rate** — TTS speed parameter; airline ~1.05x, GA ~0.95x. (Verify the TTS engine actually supports rate control without artifacts.)
  - **Optional callsign abbreviation** — GA can abbreviate to type+last-three after first call ("Cessna three alpha bravo"); airline never abbreviates.
  - **Greeting** — first call sometimes includes "good morning / afternoon" for GA. Never for airline in standard ops.
- Profile is data-driven (JSON under `Data/`) so it's easy to tune without code changes.

## Open questions

- Does operator-class data flow to the pilot speech path today? (Probably via callsign type — verify.)
- Speech rate via TTS — does Piper LibriTTS-R support it? If not, this becomes content-only (phraseology compression).
- Callsign abbreviation after first call is a real GA convention but adds parsing complexity if the controller's input (M10.2 PhraseologyMapper) has to recognize the abbreviated form. Probably scope to *output* only.
- Interaction with M11.1 (`PilotProficiency`) — orthogonal axes (proficiency × operator class) or merged? Probably orthogonal.

## Out of scope (probably)

- Foreign-carrier accents / non-native English speakers.
- Helicopter / lift / specialty operators — bucket as GA-like until they need their own profile.
- Cargo vs passenger carrier distinction (rarely audible on freq).

Lowest-risk subset to ship first: greeting + compression only, no rate change. That's pure phraseology and is the part of the deferral with the most acceptable risk. Add rate / abbreviation only after the phraseology-only version proves it doesn't generate weird readbacks.

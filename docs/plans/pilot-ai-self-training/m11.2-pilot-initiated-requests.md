# M11.2 — Pilot-initiated requests

**Status:** Stub — not yet planned. Flesh out before implementing.

## Why

Solo training today is "controller talks, aircraft obey." Real ATC has constant pilot-driven traffic: requests for lower, direct, deviation around weather, ride reports, block altitudes, speed changes for fuel. Without pilot calls the controller's workload is artificially low and one entire half of the radio is missing.

## Sketch

- New `PilotRequest` types ticked by `PilotProactive` (already exists from M10.4):
  - `RequestLower` — high-altitude cruiser asking for descent on long legs.
  - `RequestDirect` — well-positioned aircraft asking to skip a downstream fix.
  - `RequestRide` — turbulence-driven altitude change request.
  - `RequestDeviation` — weather-driven heading request (gated on the scenario actually having weather).
  - `RequestSpeed` — speed-relaxation request for fuel.
- Trigger conditions derived from aircraft state (altitude, distance to destination, time on segment) plus a small per-aircraft cooldown so the controller isn't spammed.
- Pilot speech only — the request goes into the same SAY queue as readbacks. Controller satisfies by issuing the corresponding canonical command; refusal is `STBY` or `UNABLE` (verbal, no command).
- `RpoShowPilotSpeech` already routes pilot speech to the green channel in RPO mode — same path here.

## Open questions

- Frequency budget — how often is "realistic" without being annoying? Per-aircraft cooldown + global rate cap?
- Should requests be tied to `PilotProficiency` (M11.1)? Sharp pilots ask more often, students less? Or invert that?
- Does a refused request retry? Time-out? Pilot just accepts and flies the assigned profile?
- Do we generate pilot calls when the aircraft is NOT in the student's airspace? (Probably no — only own-frequency aircraft talk to us.)
- Interaction with `PendingPilotTransmissions` queue + `FrequencyState` two-gate model — requests should respect both gates and not stomp readbacks.

## Out of scope (probably)

- Pilot bargaining / negotiation ("how about 12 thousand instead?") — single-shot requests only.
- Mid-approach requests (go-around alternatives, etc.) — let the missed-approach logic from M10.5 handle those.

## Touches

- `src/Yaat.Sim/Pilot/PilotProactive.cs` (or equivalent ticker) — new request triggers.
- `src/Yaat.Sim/Pilot/PilotResponder.cs` — request-utterance builders.
- `src/Yaat.Sim/Aircraft/AircraftState.cs` — request cooldown / state.
- `src/Yaat.Sim/Pilot/FrequencyState.cs` — gating so requests serialize with readbacks.

## Pick-up notes

`RequestLower` is the simplest first slice — pure altitude check, fixed trigger, no weather dependency. Ship one request, write the integration test, then add the others in follow-ups.

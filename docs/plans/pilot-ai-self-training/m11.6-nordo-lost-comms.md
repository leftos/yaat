# M11.6 — NORDO / lost communications

**Status:** Stub — not yet planned. Confirm scope with the user before flesh-out — this re-opens a deferred decision.

## Why

NORDO (no radio) is a real scenario controllers must handle: pilot loses comms, squawks 7600, executes lost-comms procedure per 14 CFR 91.185 (IFR) or makes light-gun calls (VFR). The training value is "what do you do when your primary tool stops working?" — diversion of nearby traffic, accommodating the lost-comms aircraft's expected behavior, coordinating with the tower / receiving fields.

## Previously deferred

The README explicitly lists **"Lost-comms 14 CFR 91.185 procedures"** as deferred indefinitely with rationale:
> Solo sessions never run that long. M10.4's pending-clearance reminders + M10.5's missed-approach autonomy cover the failure modes that matter.

Reopening this is non-trivial. The deferral logic is still mostly true — a 14 CFR 91.185-compliant lost-comms procedure takes many minutes of sim time. But a *narrow* version (pilot goes silent in the pattern; squawks 7600; player has to integrate other traffic around them) could fit a solo session and still be a genuine training event. Before implementing, the call to make is: "is the narrow version worth it, or does this stay deferred?"

## Sketch (narrow version)

- Rare event triggered randomly or on-author-flag. Aircraft squawks 7600 and stops responding to all commands.
- Aircraft continues per its **last-cleared profile** (current phase carries on; no new commands accepted):
  - If on approach → flies the approach to a landing.
  - If departing → flies the SID to the first transition fix.
  - If in cruise → continues filed route.
- Player must work around the lost-comms aircraft: vector other traffic, accept that the aircraft will land / depart per its profile.
- No `RECALL` / no `UNABLE` / no readbacks — full radio silence from that aircraft.
- Terminal annotation tags the aircraft track with `7600` (already supported in CRC datablock? — verify).

## Open questions

- **Is this worth building at all?** The deferral rationale still holds; reopening requires a clear "yes, this is worth it" from the user. If borderline, defer again and revisit later.
- Light-gun calls for VFR — out of scope or part of this? (Probably out — would need tower-cab integration.)
- How does the aircraft handle controller actions like `DROP` or `HOO` while NORDO? Probably normal — track ownership is server-side, independent of the aircraft's radio.
- Should the player be able to confirm NORDO via squawk observation (7600) before the first missed transmission, or only after they try to call?

## Out of scope (probably)

- Full 14 CFR 91.185 routing logic (last route → expected → filed, last altitude → expected → filed). Aircraft just continues current phase.
- Radio recovery — once NORDO, stays NORDO for the session.
- Pilot reports / position calls for NORDO aircraft.

# M10.1 — Pilot voice (text-only readbacks)

**Status:** shipped 2026-04-25 (commit `8e21fc7`)
**Size:** Small (~3–5 days)
**Maturity:** shipped

## Context

First production milestone of the solo-training stack. Goal: when `SoloTrainingMode` is on, every successful command dispatch produces a deterministic readback string written to `aircraft.PendingNotifications`, plus a spawn check-in for parked IFR aircraft. No new input mode (M10.2 adds NL ATC); no audio (M10.3 adds TTS). The point was to validate the readback content before adding voice or contention.

## What shipped

- **`PilotResponder.BuildReadback(CompoundCommand, AircraftState) → string?`** — inverts a parsed compound back into a pilot-readback string by walking each block's commands through `PhraseologyVerbalizer.Verbalize`. Compound conditions (`at SUNOL turn left 270`) become a leading clause attached to the first command in the block; subsequent commands suppress the lead-in.
- **`PilotResponder.BuildReadyToTaxi(AircraftState) → string`** — the spawn check-in: `"[callsign] ground, [spoken] [location], ready to taxi."` Pilot-only utterance with no controller equivalent — lives directly in `PilotResponder` rather than as an inverted `PhraseologyRule`.
- **`PhraseologyVerbalizer`** — inverts `PhraseologyRules` to produce readback strings for ~30 high-traffic commands. Single source of truth for ATC vocabulary in both directions (controller speech parsing in M10.2 will consume the same rules).
- **`SoloTrainingMode` plumbing** — `SimScenarioState.SoloTrainingMode` → `DispatchContext.SoloTrainingMode` (7th positional param) → `PhaseContext.SoloTrainingMode`. Wired at all `SimulationEngine` construction sites.
- **`AircraftGroundOps.HasAnnouncedReady`** — one-shot flag preventing the spawn check-in from re-firing each tick. Snapshot-serialized via `AircraftGroundOpsDto.HasAnnouncedReady`.
- **`AtParkingPhase.OnTick` hook** — fires `BuildReadyToTaxi` once when `SoloTrainingMode && HasFlightPlan && !HasAnnouncedReady && ElapsedSeconds >= 5`.
- **`CommandDispatcher` hook** — after a successful `DispatchCompound`, when `ctx.SoloTrainingMode`, builds the readback and appends to `aircraft.PendingNotifications`. Fires only for live user-issued commands; deferred / preset / replay paths suppressed.

## What shipped vs the original plan

The original plan (in `README.md` before the doc restructure) put `HasAnnouncedReady` on `AircraftState` directly, mirroring `IsIdenting`. Reality: it ended up on `AircraftGroundOps` because that's where ground-related per-aircraft flags already live (`IsHeld`, `GiveWayTarget`, `IsExpeditingTaxi`, `PushbackTrueHeading`). M10.1.1 keeps it where it is and places new flags by intent.

The original plan also added a `PilotPersonality.Verbatim` enum stub for future "varied" mode. That stub was deemed unnecessary at MVP and not introduced; if M10.x needs personality variants, it can introduce the enum then.

## Verification (at ship)

- ~30 golden-string tests across the most common commands.
- Round-trip test: every rule's pattern, when verbalized via a hand-constructed canonical, produces a string that re-parses to the original canonical via `PhraseologyMapper`.
- Manual: KOAK departure scenarios with `SoloTrainingMode = true` produced visible readbacks in the terminal pane for every dispatched command.

## Known limitations / follow-ups

- IFR-only spawn check-in (gated by `HasFlightPlan`). VFR aircraft don't check in. **Fixed in M10.1.1.**
- No HoldingShort / LinedUp / OnFinal check-ins — only AtParking. **Added in M10.1.1.**
- No airborne-spawn check-ins. **M10.1.2.**
- No audio. **M10.3.**
- No frequency contention; readbacks land in arbitrary terminal order. **M10.3.5.**

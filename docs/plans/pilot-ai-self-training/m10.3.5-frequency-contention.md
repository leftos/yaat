# M10.3.5 — Frequency contention + transmission queue + activity-aware verbosity

**Status:** in progress — queue foundation implemented; activity-aware verbosity still pending
**Size:** Medium (~1 week)
**Maturity:** active slice — RSP/SAY split and single-frequency serialization implemented

## Context

A sim-level frequency-state machine that serializes pilot speech, blocks pilots from talking over the controller, makes them wait for the addressee's readback before stepping in, and adapts verbosity based on how busy the frequency is. This is the layer that turns a flood of simultaneous text readbacks into the realistic one-at-a-time radio chatter players expect.

This gets its own milestone because the contention model touches several concepts that don't matter for text-only M10.1 (where everything just appends to the terminal in arbitrary order) but become essential the moment audio enters the picture in M10.3. Lifting it into its own pass keeps M10.3 focused on synthesis + DSP and lets M10.3.5 focus on the radio-protocol semantics. The active slice separates immediate command feedback from delayed radio speech: `RSP` confirms what the student's command parsed/dispatched to, while `SAY` is the delayed subtitle of what the pilot actually says on frequency.

## What ships

A frequency-state machine plus transmission queue plus activity meter. The implemented foundation serializes solo pilot speech through delayed `SAY` entries and typed pilot-audio broadcasts. Activity-aware verbosity still waits on `PilotPersonality.Varied` and populated `PilotShortcuts`.

## Concept model (per active frequency)

```
FrequencyState {
    Transmitting:        None | Controller | Aircraft(callsign)   // current air time
    AwaitingReadbackFrom: Aircraft(callsign)?                     // who owes the next transmission
    PendingTransmissions: Queue<PilotTransmission>                // pilots waiting to talk
    ActivityWindow:       rolling tx-count over last 60 s         // → Quiet | Moderate | Busy | Saturated
}
```

## State transitions

- Controller dispatches `DM 5000` to aircraft X → terminal emits immediate `RSP`; the frequency state records `AwaitingReadbackFrom = X`.
- X's readback enqueues. Fires the moment `Transmitting == None` AND `AwaitingReadbackFrom == X` → on completion both fields clear.
- Aircraft Y's proactive call (spawn check-in, pending-clearance reminder, scripted SAY output from a scenario preset) enqueues. Fires when `Transmitting == None` AND `AwaitingReadbackFrom == null`. Y waits behind X's pending readback even if Y was queued first.
- Activity meter increments on every transmission, decays at 1/60 s. Classification: `<5/min Quiet`, `5–12 Moderate`, `12–20 Busy`, `>20 Saturated`.

## What gets queued

Any solo pilot text emission. Readbacks from the dispatch hook (M10.1), spawn check-ins (M10.1+), SAY-command outputs, proactive reminders (M10.4), autonomous warnings (M10.5). These now land in `aircraft.PendingPilotTransmissions`; `SimulationWorld.ActiveFrequency` serializes them and the server/client emits delayed `SAY` plus `PilotTransmissionBroadcast`. RPO-mode SAY/readback collections remain separate.

## Verbosity gating

(Depends on a `PilotPersonality.Varied` mode + `PilotShortcuts` populated from `docs/pilot-phraseology-examples.md`.)

- `Saturated` / `Busy` → strongly prefer the shortest variant in `[verbatim] + PilotShortcuts`. Drop optional words. Avoid prefix/suffix sprinkles.
- `Moderate` → standard verbatim from the rule pattern.
- `Quiet` → may pick a verbose variant; "alright"/"thanks" personality wrappers (when populated) can fire.

## Files to touch

- `src/Yaat.Sim/Pilot/FrequencyState.cs` *(new)* — the state machine. One instance per active frequency; for MVP YAAT has effectively one (the active controller's frequency) so a single sim-level instance is enough.
- `src/Yaat.Sim/Pilot/PilotTransmission.cs` — record `(Callsign, Text, SpeechText, SourceKind, Kind)` where `Kind ∈ { Readback, Proactive, Report, SayReadback }`.
- `src/Yaat.Sim/Pilot/FrequencyActivityMeter.cs` *(new)* — rolling-window counter with classification.
- `src/Yaat.Sim/Pilot/PilotResponder.cs` — routes solo speech into `PendingPilotTransmissions` rather than immediate `PendingNotifications`.
- `src/Yaat.Sim/SimulationWorld.cs` — owns `ActiveFrequency`, records expected readbacks, drains ready solo pilot transmissions.
- `src/Yaat.Sim/Simulation/SimulationEngine.cs` and `yaat-server` tick processing — drain ready transmissions after physics and emit delayed `SayPilot` / `SayReadback` terminal entries plus `PilotTransmissionBroadcast`.
- Command dispatch / server command handling — on successful solo controller dispatch, set `AwaitingReadbackFrom = aircraft.Callsign`; the first matching readback clears it.

## Riskiest parts

1. **Audio duration estimation.** The current foundation estimates airtime from word count so deterministic text/SAY ordering works before consuming exact synthesized duration. A later audio-tightening pass can feed measured duration back into the queue.
2. **Starvation.** A pilot with low priority + a permanently busy frequency could wait forever. Add max-wait → drops the transmission with a debug log, never silently.
3. **Multiple frequencies.** Today's simplification is one frequency; if YAAT eventually models per-facility frequencies (Tower vs Ground vs Approach), each gets its own state. Architect for this from day one — `Dictionary<FrequencyId, FrequencyState>`.
4. **Tick determinism.** Replays must produce identical chatter ordering. The activity meter's rolling window must be tick-deterministic, not wall-clock.

## Verification

- Unit test: enqueue a proactive check-in for aircraft Y while controller awaits X's readback. Assert X's matching readback drains first even if Y was queued first.
- Unit test: simulate 30 pilot transmissions in the rolling window. Assert activity meter classifies as `Saturated`.
- Focused integration: successful solo commands keep immediate `RSP`; pilot readbacks drain later as `SayPilot` / `SayReadback`.
- Replay test: record 60 s of mixed traffic, replay, confirm transmission ordering is bit-identical.

## Why not in M10.1

M10.1's job was "verify the readback content is right." Adding contention there would have exploded the test surface (timing, ordering, queue starvation) before the *strings* were validated. M10.3.5 refactors the solo pilot-speech call sites to `PendingPilotTransmissions` plus `FrequencyState` drain, preserving phraseology while separating immediate `RSP` from delayed `SAY`.

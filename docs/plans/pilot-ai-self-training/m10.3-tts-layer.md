# M10.3 — TTS layer (audible pilot voice)

**Status:** implemented v1
**Size:** Medium
**Maturity:** shipped in code — revalidated and implemented on 2026-05-07

## Context

M10.3 adds an audible pilot-voice side channel for solo training. The current implementation is event-driven and keeps the sim free of platform audio dependencies:

1. `Yaat.Sim` records typed pilot transmissions alongside the existing terminal queues.
2. `yaat-server` drains those typed transmissions after terminal broadcasts and sends a SignalR `PilotTransmissionBroadcast` event.
3. `Yaat.Client` gates the event by user preference, active scenario id, and active `SessionSoloTrainingMode`, then queues it through the local pilot voice service.

Terminal text remains the source of truth. The typed audio event carries the exact same text that the terminal shows.

## What shipped

- New sim-side `PilotTransmission` queue on `AircraftState`.
- `PilotResponder.QueueSoloPilotTransmission(...)` and `QueueSoloPilotReadback(...)` mirror existing `PendingNotifications` / `PendingPilotReadbacks` entries into the typed queue without changing phraseology.
- `SimulationWorld.DrainAllPilotTransmissions()` drains the side queue for server broadcast.
- New server/client DTO contract:

```csharp
PilotTransmissionBroadcastDto(
    string ScenarioId,
    string Callsign,
    string Text,
    string SourceKind,
    int SpeakerId,
    double ElapsedSeconds,
    DateTime Timestamp)
```

- New SignalR event: `PilotTransmissionBroadcast`.
- New client event: `ServerConnection.PilotTransmissionReceived`.
- New server interface method: `ITrainingBroadcast.BroadcastPilotTransmission(...)`.
- Deterministic server-side speaker assignment: `PilotVoiceAssigner.AssignSpeakerId(scenario.RngSeed, callsign)` returns `0..903`.
- Client preferences, default off:
  - `PilotVoiceEnabled`
  - `PilotVoiceVolume` default `80`
  - `PilotVoiceRadioFxEnabled` default `true`
- Settings UI adds a Pilot Voice section under Speech.
- `PilotVoiceService` queues transmissions FIFO and speaks them one at a time through sherpa-onnx + PortAudio.

## Current Engine

The production path uses the same cross-platform stack validated in M10.0:

- `org.k2fsa.sherpa.onnx` for Piper VITS synthesis.
- Piper `en_US-libritts_r-medium` voice directory, installed on demand from Settings -> Speech -> TTS.
- `NAudio.Dsp` for radio filtering and squelch tail.
- `PortAudioSharp2` for cross-platform output playback.

The client remains silent when the voice pack is not installed or no output device is available, and the feature is off by default. The voice pack is discovered from `.tmp/voices/vits-piper-en_US-libritts_r-medium` during development or `%LOCALAPPDATA%/yaat/voices/vits-piper-en_US-libritts_r-medium` for app data. Settings exposes the Piper download/remove flow next to the TTS controls.

## Routing Rules

Server emits `PilotTransmissionBroadcast` only for solo-training pilot transmissions:

- Solo readbacks generated after accepted student commands.
- Solo ground/airborne/pattern check-ins and reminders that already appear as terminal `Response`.
- Solo RTIS/RFIS readbacks routed through the SAY channel.
- SAY-class terminal entries emitted by the command dispatcher while `SoloTrainingMode` is active.

Server does not emit typed audio for generic warnings, normal terminal responses, or RPO-mode `PilotSpeech`.

# M10.1.5 — VFR airspace boundary respect (Class B / Class C)

**Status:** planned (sketch)
**Size:** Medium (~3–5 days, depending on what airspace data is already available)
**Maturity:** sketch — needs exploration

## Context

Today a VFR aircraft happily crosses any controlled-airspace boundary without
asking. In real flight that's a violation: Class B requires an explicit
clearance ("cleared into the Class Bravo") before entry, Class C requires
established two-way radio communications (the controller has used the
aircraft's callsign in a transmission) before entry, and Class D requires the
same two-way comms but inside its smaller cylinder. M10.1.5 makes the pilot
AI honor those boundaries autonomously so the student has to actually issue
the clearance / acknowledge the call before traffic enters their airspace.

This pairs naturally with the M10.1.x VFR-emphasis push: M10.1.2 ships the
airborne check-in, M10.1.3 ships VFR pattern speech, and M10.1.5 ships the
behavioral consequence of *not* yet having two-way comms / clearance.

## What ships (target outcome)

VFR aircraft inbound on a track that would cross a Class B or Class C
boundary self-restrict at the boundary instead of penetrating. Behavior is
deterministic and observable:

- **Approach detection.** As an aircraft's projected ground track approaches
  a controlled-airspace boundary at altitudes that fall inside the airspace's
  vertical limits, evaluate the gate.
- **Class B gate.** Has the controller issued the explicit *cleared into the
  Bravo* command? If not → hold outside. (New canonical command needed for
  the explicit clearance: probably `CB` / `CLBRV` or similar — see M10.5's
  `unable` discussion for the analogous pattern.)
- **Class C / D gate.** Has the aircraft established two-way comms with the
  student's facility? Defined as: `HasMadeInitialContact == true` *and* the
  controller has used this aircraft's callsign in at least one transmission
  (any command dispatched against it counts). If not → hold outside.
- **Self-restrict behavior.** Slow to maneuvering speed and turn parallel to
  the boundary, or orbit if there's no clean parallel track. Pilot
  transmits: `"[callsign] holding outside the bravo, [N] miles [direction]
  of [airport-spelled], awaiting clearance."` (Class C analog: `"...holding
  outside the charlie, awaiting two-way."`)
- **Resume on satisfaction.** When the gate flips (controller issues the
  clearance / acknowledges by callsign), the pilot AI resumes the original
  course on the next tick.
- **Solo-training-mode gated.** Instructor-mode users get zero behavior
  change.

## Architecture (rough)

Probably a new pilot-side phase `RespectAirspaceBoundaryPhase` that takes
over `Targets` while the gate fails. Inserts itself ahead of the aircraft's
existing nav phase when entry is imminent (~30–60 s of projected time to
boundary), and removes itself when the gate clears. Trigger and clearance
logic live in `PilotProactive` (where M10.5's autonomous contingencies will
also live).

A controller-acknowledgement hook is needed: every successful
`CommandDispatcher.DispatchCompound` against an aircraft sets
`AircraftState.HasReceivedControllerCallsignUse = true` (or similar — name
TBD). That's the "two-way comms established" flag for Class C/D.

For Class B, a dedicated canonical command (`CLBRV` "cleared into the
bravo"?) sets `AircraftState.IsClearedIntoBravo = true`. Wording follows
AIM 4-3-1 / AIM 3-2-3.

## Needs exploration

- [ ] **Where do Class B / Class C / Class D polygons live in YAAT today?**
  Best guess: nowhere yet, since no feature has needed them. May need to
  parse from vNAS data (the ARTCC config?) or from a new ground-data
  artifact. If absent, the milestone's blocker is *getting* the airspace
  data, not the pilot-AI logic.
- [ ] **Does the YAAT scenario format have a primary-airspace-class hint?**
  If the student is at `OAK_TWR` we can assume Class C around OAK; if at
  `SFO_TWR` we assume Class B. But that's a sloppy proxy — better to read
  airspace polygons directly.
- [ ] **What's the right "imminent entry" trigger distance?** Projecting
  ground track 30 s ahead and checking if it crosses the polygon, or just
  testing aircraft-position-vs-polygon with an outer buffer ring (~2 nm)?
  Buffer ring is simpler and probably good enough.
- [ ] **Hold-outside behavior detail.** Just orbit? Turn 90° away from the
  boundary? Slow to 90 KIAS and S-turn? `aviation-sim-expert` should weigh
  in on what real pilots do. AIM 3-2-3 / 3-2-4 are the anchors.
- [ ] **Class B explicit-clearance command.** New canonical needed
  (`CLBRV` / `CCLEARED`?). Verify AIM 4-1-21 phraseology for the
  controller's clearance form: *"[callsign], cleared into the Bravo"* or
  *"[callsign], cleared through the Bravo"*. Wire into
  `PhraseologyRules` so M10.2 student-input mapping picks it up.
- [ ] **Two-way-comms acknowledgement detection.** AIM 4-1-20 makes "two-way
  radio communications established" specific: the controller has used the
  aircraft's callsign in a response (not a generic "all aircraft"
  transmission). Any successful dispatched command suffices. Generic SAY
  responses to nobody don't.
- [ ] **Edge: aircraft already inside the airspace at scenario spawn.** A
  scenario that places a VFR aircraft already inside Class B without
  clearance shouldn't trigger the AI to immediately panic-divert.
  Initial-spawn aircraft should be treated as already-cleared (or at least
  not retroactively boundary-restricted). Scenario authors are responsible
  for the realism of their spawn placement.
- [ ] **Class E transitions.** Class E (controlled but no clearance
  required for VFR) is out of scope — VFR aircraft can enter freely. M10.1.5
  only covers B/C/D.
- [ ] **VFR-on-top, Special VFR, etc.** Out of scope. M10.1.5 is the
  baseline rule; corner cases defer indefinitely unless a real scenario
  forces them.

## Riskiest part

**Sourcing airspace polygons.** If YAAT has to ingest a new data file, the
real cost is in data plumbing and validation, not pilot AI. If vNAS already
publishes Class B/C boundaries (the messaging package, an ARTCC config
field, or a separate boundary file), the milestone is small. If not, this
becomes "research the canonical source for US controlled airspace
boundaries, integrate it, validate against a reference field" before we
even get to the autonomy work.

Second-riskiest: the *controller-acknowledged* signal for Class C. Two-way
comms is a real-life soft criterion — controllers and pilots agree it's
established the moment the controller uses the callsign. Mapping that to
"any successful dispatch" should be safe but `aviation-sim-expert` should
sanity-check.

Manual:

   aircraft 10 nm south at 3,500 ft inbound for transition through Class B.
   At ~2 nm from boundary, the aircraft should slow and orbit, with a
   pilot transmission *"...holding outside the Bravo, awaiting clearance."*
   Issue the new "cleared into the Bravo" command — the aircraft resumes
   course and enters the airspace.
2. KOAK Class C scenario: spawn a VFR aircraft inbound. Without ever
   issuing a command against the aircraft, it should hold outside Class C.
   Issue any command (e.g. `FH 270`), and the aircraft resumes inbound.
3. Instructor mode (`SoloTrainingMode = false`): same scenarios, no
   self-restriction — instructor controls everything via PTT.
4. Aircraft spawned already inside Class B without clearance: no panic
   divert. The scenario author owns realism.

## Open questions

- Should the same logic apply to **IFR** aircraft? IFR has its own
  clearance lifecycle (AIM 5-2-1 ff.) — typically the IFR clearance issued
  on the ground covers the entire route through Bravo airspace. M10.1.5
  scope is VFR-only; IFR airspace-respect would be a separate milestone if
  ever needed.
- Should there be a **"cleared through the Class Bravo, maintain VFR"**
  command, or just one generic `CLBRV`? Probably one canonical with an
  optional "through" / "into" modifier — defer the modifier until M10.2
  shows what controllers naturally type.
- **Departure clearances** out of Class B/C airports for VFR aircraft —
  same gate? Separate? AIM 4-1-21 says VFR departures from a Class B
  primary need a clearance just like arrivals. The pilot-AI side at the
  spawn-on-ramp moment doesn't need extra logic if the existing
  ground-clearance flow is rigorous.

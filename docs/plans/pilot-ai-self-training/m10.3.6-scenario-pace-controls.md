# M10.3.6 - Scenario-load training pace controls

**Status:** planned
**Size:** Small-Medium (~2-3 days)
**Maturity:** drafted

## Context

Many training files are authored with RPO pace control in mind: parked aircraft may be present at load time, but a human RPO can decide when each aircraft calls Ground. Solo-training needs the same pacing control so a student can start with a slower workload and gradually increase difficulty without editing the scenario file.

M10.3.6 adds a scenario-load training setup control for how often aircraft spawned at parking make their initial ready-to-taxi call. This is not a realism rule; it is a training-rate control over a real initial-contact behavior. The underlying call remains based on AIM 4-2-3 initial contact format: facility name, full aircraft identification, surface position, and request. AIM 4-2-2 also supports not jamming a busy frequency and waiting before calling again, which is the training rationale for suppressing some initial calls at lower difficulty.

## What ships

- The existing difficulty overlay becomes a shared **Scenario Setup** modal. It appears only when it has something to ask: multiple scenario difficulties, or solo-training pace controls.
- Add a `Parking initial call-up rate` slider to that modal when solo-training pace controls are active.
- Slider range: `0%` to `100%`, step `5%`, default `100%`.
- The selected value applies to the loaded session/scenario and is remembered as the next default. Settings may expose the remembered default later, but the primary UX is scenario load.
- Auto-loaded scenarios keep the existing behavior by selecting the hardest difficulty and `100%` call-up rate unless a future command-line option explicitly overrides it.

## Implementation shape

- Add a synced and replayable session setting named `SoloParkingInitialCallupRatePercent` (or equivalent if the implementation uses the repo's current naming convention). Default is `100`.
- Carry the setting through client preferences, the scenario-load modal view model, client/server session settings DTOs, `SimScenarioState`, snapshots/replay, and server broadcasts.
- `AtParkingPhase` uses the setting before queuing the 5-second ready-to-taxi check-in:
  - `100` means current behavior: all eligible parked aircraft call.
  - `0` means no parked aircraft auto-call.
  - intermediate values select a deterministic subset using a stable hash of scenario identity plus aircraft callsign. Do not use process-randomized `HashCode` for this.
- When an aircraft is suppressed by the rate gate, record that the parking auto-call decision has been processed so it does not retry every tick. Do not mark `HasMadeInitialContact` for suppressed aircraft.

## Files to touch

- `src/Yaat.Client/ViewModels/MainViewModel.Scenario.cs` and `src/Yaat.Client/Views/MainWindow.axaml` - rename/generalize the difficulty overlay into the shared scenario setup modal and add the slider.
- `src/Yaat.Client.Core/Services/UserPreferences.cs`, `ServerConnection.cs`, and matching server DTO/hub/session state - persist the last chosen default and sync the per-session value.
- `src/Yaat.Sim/Simulation/SimScenarioState.cs`, `PhaseContext.cs`, and `src/Yaat.Sim/Phases/Ground/AtParkingPhase.cs` - carry the setting into the sim and apply the deterministic rate gate.

## Verification

- Client tests: modal appears for difficulty+slider, difficulty-only, slider-only, and neither; Load/Cancel behavior stays consistent with the current difficulty prompt.
- Preference/session tests: selected call-up rate is remembered, sent after scenario load, broadcast to other clients, and replayed.
- Sim tests: `100%` preserves current parking check-in behavior; `0%` suppresses all parking auto-calls; a middle rate is deterministic across reload/replay.
- Manual: load a solo-training scenario with multiple parked aircraft at `0%`, `50%`, and `100%`; confirm the number of initial Ground calls follows the selected pace without changing the scenario file.

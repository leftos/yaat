# M10.1.2 — Airborne-spawn check-ins (IFR + VFR)

**Status:** planned (sketch)
**Size:** Small–Medium (estimate after exploration)
**Maturity:** sketch — needs exploration

## Context

A VFR aircraft inbound for full-stop calls Tower / Approach from 5–10 NM out, well before reaching `FinalApproachPhase`. Without an airborne-spawn check-in, a VFR inbound stays silent until on final, which doesn't match real ATC. Same holds for IFR airborne arrivals (handed off mid-cruise) and VFR overflight aircraft transiting the student's airspace.

This was originally folded into M10.4's "Pilot expectations + proactive requests" milestone, but the user has called for VFR emphasis up front, so airborne-spawn check-ins move ahead of M10.2/M10.3 to land alongside M10.1.1's ground-spawn work.

## What ships (target outcome)

When an aircraft is first ticked airborne and `!HasMadeInitialContact && SoloTrainingMode`, fire a check-in template chosen from the `Scenario.StudentPositionType` × `aircraft.FlightPlan.IsVfr` grid. Plus a generic sign-off when an aircraft is handed off via HOO ("frequency change approved" → pilot acknowledges and leaves frequency).

Tentative template grid (subject to AIM 4-2-3 / 5-5-1 verification with `aviation-sim-expert`):

| Student position | IFR | VFR (full-stop or transit) |
|---|---|---|
| Tower (Class D) | Rare — direct-to-tower IFR. "Tower, [callsign] runway [rwy], with information Alpha." | "Tower, [callsign] [N] miles [direction] at [altitude], inbound for landing, with information Alpha." (full-stop) / "Tower, [callsign] [N] miles [direction] at [altitude], request transition [direction]." (overflight) |
| Approach (TRACON) | "Approach, [callsign] level [altitude], with information Alpha." | "Approach, [callsign] [N] miles [direction] at [altitude], request [landing/transition], with information Alpha." |
| Center (ARTCC) | "Center, [callsign] flight level [FL]." | "Center, [callsign] at [altitude], [N] miles [direction] of [airport]." |
| Ground (rare for airborne) | skip | skip |

Direction is computed from aircraft position relative to the primary airport using `GeoMath.DistanceNm` plus an 8-point compass quantizer. Altitude renders via existing `AltitudeToWords` helper (sub-FL180) or "flight level X" (FL180+).

## Needs exploration

Before this becomes active:

- **VFR overflight intent inference.** "Request transition northbound" requires knowing the aircraft's direction of travel and whether the student's airport is the destination or just airspace they're crossing. Source candidates: `aircraft.FlightPlan.Destination` (if filed), heading at first airborne tick, or a new `Scenario.AircraftIntent` field. Need to decide before writing templates.
- **Negative flight following.** A VFR aircraft with no destination filed and no service request has different phraseology — closer to "Tower, Cessna 12345, VFR, request..." Need an explicit signal in the scenario format or aircraft state to distinguish. May require a new field.
- **First-airborne-tick detection.** `IsOnGround` flips false on takeoff. We want this to fire *only* for aircraft that spawn airborne, not for aircraft that took off in-session. Today `AtParkingPhase` sets `HasMadeInitialContact = true` (per M10.1.1) which gates this — but verify there's no other ground-only path that misses the flag set.
- **HOO sign-off template + trigger.** Where in the HOO command path do we emit a pilot acknowledgment? Probably the HOO command handler itself. Need to find it (`tools/Yaat.LayoutInspector` won't help — search `CommandDispatcher` / handlers for `Handoff`).
- **Generic sign-off vs branched.** "[callsign], [next-frequency], [callsign], so long" is a single template. If the next frequency isn't known (handed off to UNICOM with no further service), drop the frequency: "[callsign], so long, frequency change approved." That branch needs the upstream HOO to expose whether a next frequency exists — verify.
- **`Scenario.StudentPositionType` access from a sim tick.** Confirm the field exists, where it lives (probably `Scenario` or `SimScenarioState`), and the access pattern from a per-aircraft tick. This drives the grid lookup.

# M12.2 — Phraseology grading

**Status:** Stub — not yet planned. Flesh out before implementing.

## Why

FAA 7110.65 prescribes specific phraseology for almost everything: element order (callsign first, type of clearance, then instruction), "niner" for nine, "decimal" / "point" for decimals, altitude phrasing ("five thousand" not "fifty hundred"). Today YAAT validates that a command parses to a canonical form, but doesn't grade *how* it was issued. A controller can rack up a perfect safety score while sounding like nobody who's ever held a license — and not know it.

## Sketch

- Per-transmission grading pass on every controller utterance (typed or spoken via M10.2). Runs after the command parses successfully.
- Each transmission gets a score (0-100) and zero or more findings. Findings cite the 7110.65 paragraph.
- Initial rule set (small, expand later):
  - Element ordering — callsign first.
  - "Niner" for 9 in any numeric (currently the controller side doesn't enforce; M10.3.5 / pilot side already say "niner").
  - "Decimal" or "point" between integer and fractional frequency components.
  - Altitude phrasing — "five thousand five hundred" not "five thousand and five hundred"; never "fifty-five hundred."
  - Heading phrasing — three digits always ("heading zero niner zero" not "heading ninety").
  - Runway phrasing — "two eight right" not "twenty-eight right."
- Live mode (during session): findings post to a side panel, soft warnings only. Aggregated mode (Session Report): full breakdown by transmission.

## Open questions

- Source of canonical pronunciation rules: 7110.65 chapter 2 mostly. Codify as a JSON rule file?
- Voice input (M10.2) — does the STT preserve enough of the user's actual words to grade phraseology, or does the PhraseologyMapper normalize too aggressively? May need to capture both the raw STT transcript and the canonical command for grading.
- Typed input — there's no phraseology to grade if the user types `N12345 H180`. Should typed commands skip phraseology grading entirely? Or grade what the canonical *should* sound like when spoken? Probably skip — phraseology is a spoken concept.
- Does this become annoying? Lots of small dings = noise. Maybe: only flag in Session Report; don't show live unless user opts in.

## Out of scope (probably)

- Cadence / pace / volume grading (TTS / mic-quality territory, not phraseology).
- Foreign phraseology (ICAO Annex 10 differences from 7110.65).
- Grading the *pilot* side (we control that anyway).
- Tactical phraseology vs. strict 7110.65 form ("turn 20 right" vs. "right turn heading two seven zero" both ship in real ops). Grade only the elements that are unambiguous, defer the gray-area ones.

## Touches

- `src/Yaat.Sim/Phraseology/PhraseologyRules.cs` — already the source of truth for parsing; extend with a *generator* side that can grade.
- `src/Yaat.Sim/Session/` — new phraseology-finding type.
- Session Report UI — per-transmission breakdown.
- Possibly `src/Yaat.Sim/Speech/` — to access raw STT transcript pre-normalization.

## Pick-up notes

Start with three rules only (niner, three-digit headings, decimal in frequencies) and ship them as Session Report findings only — no live display. That's enough to prove the grading loop. Expand the rule set after the loop is solid; the rule list itself is the easy part.

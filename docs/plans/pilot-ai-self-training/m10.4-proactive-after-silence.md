# M10.4 — Pilot expectations + pending-request reminders

**Status:** planned
**Size:** Medium (~4–6 days; was originally Small-Medium before pending-request tracking was added)
**Maturity:** drafted — needs revalidation when active. **Trimmed**: the airborne-arbitrary-spawn check-in grid moved to M10.1.2 as part of the VFR-emphasis pull-forward.

## Context

A solo student who forgets to clear an arriving aircraft for the approach gets nothing today — the aircraft just keeps flying its last clearance silently. Real ATC includes pilot prompts: "United 123, with you, ten miles to land 28R" pings the controller.

M10.4 introduces a tiny `PilotExpectation` value object on `AircraftState` (NOT a full intent system) plus a pending-request tracker for pilot-originated requests/check-ins. The tracker lets the pilot recognize that the last request has not been handled and call again after an appropriate delay. A targeted `STBY` / `ROGER` counts as acknowledgement, but not satisfaction: per 7110.65 2-1-18, `STAND BY` is not approval or denial and the controller will respond later. That means the pilot waits much longer before following up after standby than after no response at all.

FAA grounding:

- AIM 4-2-2: pilots should listen before transmitting and wait briefly before calling again because the controller may be writing, transmitting elsewhere, or selecting another transmitter.
- AIM 4-2-3: initial/subsequent contact should identify the facility, aircraft, surface position when applicable, and the request.
- AIM 4-4-7: acknowledgements/readbacks include aircraft identification so the controller knows the correct aircraft received the instruction.
- 7110.65 2-1-18: operational requests are approved, unable, or told to stand by; standby acknowledges the request but is not approval or denial.

## What ships

- A small `PilotExpectation` record on `AircraftState` computed each tick from current phase + flight-plan state + clearance state.
- A small `PilotPendingRequest` state on `AircraftState` for the latest pilot-originated request/check-in that expects controller action.
- A `PilotProactive` ticker that fires reminders in the scenarios below.
- Richer phraseology in handoff acknowledgments (next frequency from `PilotExpectation`).

The airborne-arbitrary-spawn check-in grid that originally lived here ships as part of M10.1.2.

## Files to touch

- `src/Yaat.Sim/Pilot/PilotExpectation.cs` *(new)* — small records:
  ```csharp
  public sealed record PilotExpectation
  {
      public ExpectationKind Kind { get; init; }   // OnRamp, ReadyForTaxi, HoldingShort,
                                                   // EnRouteFiledRoute, ApproachExpected,
                                                   // EstablishedOnApproach, LandedTaxiingIn
      public long? UnsatisfiedRequestSinceMs { get; init; }
      public string? PendingRequestKind { get; init; }  // "taxi", "takeoff", "landing"
  }

  public sealed record PilotPendingRequest
  {
      public PendingPilotRequestKind Kind { get; init; }  // Taxi, Takeoff, Landing, Approach, AirspaceEntry
      public PendingPilotRequestResponseState ResponseState { get; init; }  // None, Standby, Satisfied, Denied, Superseded
      public double FirstRequestedAtSeconds { get; init; }
      public double LastRequestedAtSeconds { get; init; }
      public double NextFollowUpDueSeconds { get; init; }
  }
  ```
  Snapshot-serialized. `PilotExpectation` is computed each tick by `ExpectationUpdater` from current phase + `HasFlightPlan` + `ActiveSidId` / `ActiveStarId` / `DestinationRunway` + clearance state (`DepartureClearance`, `LandingClearance`, `ActiveApproach`). `PilotPendingRequest` is updated when pilot-originated requests/check-ins are emitted and when controller commands respond to them.

- `src/Yaat.Sim/Pilot/PilotProactive.cs` *(new)* — ticked from `SimulationEngine` post-physics:
  - **Holding short waiting for takeoff with no `DepartureClearance`**: push a "ready for departure 28R, callsign" reminder.
  - **Inside 10 nm of destination on a STAR with no `ActiveApproach`**: push a "with you, ten miles to land 28R" check-in.
  - **At parking with no taxi/push clearance after the initial ready-to-taxi call**: push "ready to taxi" again.
  - **VFR controlled-airspace entry request / boundary hold pending**: if the pilot has requested entry or reported holding outside and no satisfying command follows, repeat the request/report on the same pending-request schedule.

- `src/Yaat.Sim/Pilot/PilotRequestTracker.cs` *(new or folded into `PilotProactive` if small)* — records and resolves pending pilot requests:
  - Record only pilot-originated requests/check-ins, not command readbacks or other pilot responses.
  - `None`: no targeted controller response yet; schedule a normal follow-up by request kind, generally 60-120 s.
  - `Standby`: targeted `STBY` / `ROGER` acknowledges receipt but does not satisfy the request; schedule a later follow-up, generally 3-5 min.
  - `Satisfied`: a matching taxi/push, takeoff, landing, approach, airspace, or frequency/change command closes the request.
  - `Denied`: `UNABLE`, remain-outside, or equivalent denial closes or transforms the request.
  - `Superseded`: a newer pilot request replaces the prior pending request.

- `src/Yaat.Sim/Pilot/PilotResponder.cs` — extend templates to use `PilotExpectation` for richer phraseology (e.g., handoff acknowledgment includes the next frequency from `ExpectationUpdater`).

- `src/Yaat.Sim/Commands/CommandDispatcher.cs` and command handlers that bypass it — mark matching pending requests as `Standby`, `Satisfied`, or `Denied` when live controller commands respond to the pilot.

- `src/Yaat.Sim/Simulation/SimulationEngine.cs` — call `ExpectationUpdater.Update(ac)` and `PilotProactive.Tick(ac)` after physics, before broadcast.

## Riskiest part

Spamming the terminal. Each proactive request must have a per-request timestamp gate so it fires once per situation, not once per tick. Aggressive cap: a single aircraft cannot generate more than 1 proactive line per 30 s, and a standby-follow-up must be substantially later than a no-response follow-up.

The other risk is confusing acknowledgement with satisfaction. `STBY` / `ROGER` should still satisfy M10.1.5's Class C two-way-comms gate, but it must not close the pending taxi/takeoff/landing/approach/airspace request unless a satisfying command follows.

## Verification

- New scenario test: aircraft at parking with flight plan, no commands issued → after 5 s a check-in line appears (via M10.1's AtParking; no new behavior here), after 125 s a second M10.4 reminder, no third reminder for another 120 s.
- Pending-request tests:
  - No targeted response after a pilot request schedules the normal follow-up.
  - Targeted `STBY` / `ROGER` schedules a later follow-up and does not close the request.
  - A satisfying command closes the request and prevents follow-up.
  - A readback generated by a controller command does not create a pending request.
  - `STBY` / `ROGER` still satisfies the Class C two-way-comms gate while leaving the underlying request pending.
- Aviation review by `aviation-sim-expert` that the prompts are AIM-realistic.

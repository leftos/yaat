**Status:** complete

## Context

Once the simulation produces realistic readbacks, accepts natural-language ATC, speaks aloud, and contains autonomous pilot behaviors (M10.1 → M10.5), the remaining user-facing piece is documentation that explains how to use solo training with real ARTCC-authored or local scenarios.

YAAT should **not** create training scenarios from scratch as part of this milestone. Scenario content belongs to the training staff of each ARTCC; YAAT's responsibility is to make solo-training behavior discoverable, explain the controls, and document what a scenario author can optionally include to make an existing scenario work well in solo mode.

## What ships

- `USER_GUIDE.md` section "Solo Training" that explains:
  - enabling Solo Training Mode and optional Pilot Voice;
  - loading existing ARTCC or local scenarios;
  - using the scenario-load pace controls for parked-aircraft call-ups and arrival generators;
  - what pilot readbacks, `SAY` transcripts, pending-request reminders, airspace holds, DA/MDA go-arounds, and `unable` responses look like;
  - the difference between immediate command `RSP` feedback and delayed pilot radio `SAY` traffic;
  - how `STBY` / `ROGER` affect pending requests and Class C two-way communications.
- Scenario-author notes that describe what existing scenarios should include to work well in solo training (parking spawns, arrival generators, expected approaches, handoff/frequency data, useful difficulty variants) without inventing new scenario files.
- Cleanup confirmation that the old superseded `docs/plans/pilot-ai-architecture.md` is absent. Per CLAUDE.md ("No phantom features. Replace, don't deprecate."), it should not return as a parallel source of truth.

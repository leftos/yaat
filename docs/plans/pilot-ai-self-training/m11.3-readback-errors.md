# M11.3 — Realistic readback errors

**Status:** Stub — not yet planned. Flesh out before implementing.

## Why

In real ops, pilots occasionally hear wrong, read back wrong, or omit elements — and the controller is responsible for catching the mistake and re-issuing. Today every readback in solo mode is verbatim correct, which trains the controller to *not* listen. Readback monitoring is one of the most important controller skills; it's silly that we don't exercise it.

## Sketch

- Low-rate (~2-5% per transmission, tunable) perturbation applied **after** `PilotResponder` builds the canonical readback:
  - **Drop an element** — read back heading but not altitude, or vice versa.
  - **Wrong numeric** — altitude misheard by 1000 ft, frequency off by one digit.
  - **Wrong runway side** — `28L` instead of `28R`.
  - **Misheard callsign** — read back as a similar-sounding flight (e.g., `AAL235` → `AAL237`). Forces "I say again" workflow.
- Aircraft's *internal* state (`AssignedHeading`, `AssignedAltitude`, etc.) follows the **bad readback**, not the original command. If the controller catches it and re-issues, the aircraft re-readbacks and follows the new value. If the controller doesn't catch it, the aircraft flies the wrong number — which the scoring system (M12) flags as a missed readback.
- Tied to `PilotProficiency` (M11.1): `Student` higher rate, `Sharp` near zero.
- Gated by a global toggle so authors / users can disable for clean drills.

## Open questions

- How does this interact with `CommandDispatcher`'s validation? The command was valid; the readback diverges. We probably can't piggyback on dispatch — the divergence has to happen during readback construction and then **rewrite** what gets applied to `AircraftState`.
- Should the SAY transcript display the *wrong* readback verbatim? (Yes — otherwise the controller has nothing to catch.)
- Do we need a "controller says 'I say again'" workflow command? Or does re-issuing the original command implicitly correct?
- Scoring interaction (M12): a missed readback that the controller doesn't catch is graded how? Half-credit because the pilot misheard? Full ding because the controller is ultimately responsible?
- Test determinism — same answer as M11.1: pin error rate to 0 in tests; opt-in fixtures explicitly enable.

## Out of scope (probably)

- Bad readbacks on coordination calls (no pilot involved).
- Multi-element errors (one mistake at a time keeps the catch tractable).

## Touches

- `src/Yaat.Sim/Pilot/PilotResponder.cs` — perturbation pass after readback build.
- `src/Yaat.Sim/Commands/CommandDispatcher.cs` — separate the "what we accepted" from "what the aircraft now believes."

## Pick-up notes

The hardest part is the state-divergence model — make sure the canonical command and the aircraft's resulting state can disagree. Start with one error mode (wrong altitude) and prove the loop end-to-end (perturb → state diverges → controller re-issues → state corrects → SAY-line history shows both readbacks). Don't expand error types until the loop works cleanly.

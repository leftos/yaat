# M10.1.3 — VFR closed-traffic pilot speech

**Status:** planned (sketch)
**Size:** Small–Medium (estimate after exploration)
**Maturity:** sketch — needs exploration

## Context

VFR closed-traffic / pattern work is a core training scenario for Class D Tower work. A pilot announces intent on initial contact ("request closed traffic"), then reports each pattern leg as the aircraft cycles (downwind / base / final). YAAT already has the phase plumbing (`PatternEntryPhase`, `DownwindPhase`, `BasePhase`, `UpwindPhase`, `CrosswindPhase`, `VfrFollowPhase` under `src/Yaat.Sim/Phases/Pattern/`) and `FinalApproachPhase` already detects pattern traffic via `_isPatternTraffic`. Only pilot speech is missing.

User confirmed scope on 2026-05-02: full pattern realism — initial-call closed-traffic request *plus* per-leg announcements.

## What ships (target outcome)

- **Initial-call closed-traffic request** — fired on the first pattern phase entry (likely `PatternEntryPhase.OnStart` or first relevant pattern phase): `"[callsign] tower, [spoken] [N] miles [direction], inbound, request closed traffic, with information Alpha."` Sets `HasMadeInitialContact = true`.
- **Per-leg announcements** on `OnStart` of each pattern phase, gated by `IsVfr && SoloTrainingMode`:
  - `DownwindPhase` → `"[callsign] downwind runway [rwy]."`
  - `BasePhase` → `"[callsign] base runway [rwy]."`
  - `FinalApproachPhase` first tick when `_isPatternTraffic` → `"[callsign] final runway [rwy]."` (suppress the M10.1.1 OnFinal check-in for pattern traffic — leg report is the right speech)
- Each leg fires once per phase entry. Pattern phases create fresh instances per lap, so phase-instance state suffices (mirroring `HoldingShortPhase`'s `_hasAnnouncedReady`).

## Needs exploration

Before this becomes active:

- **First pattern phase identity.** Which phase is entered first when a VFR aircraft is cleared into closed traffic? `PatternEntryPhase`? `DownwindPhase`? Depends on entry direction (45° entry, midfield crosswind, etc.). Need to read `PatternBuilder.cs` / `PatternEntryPhase.cs` to see how the phase machinery sequences.
- **Suppressing the M10.1.1 OnFinal check-in for pattern traffic.** Pattern aircraft enter `FinalApproachPhase` via the pattern path (with `_isPatternTraffic = true`). They should *not* fire M10.1.1's spawn-on-final check-in (that's for aircraft that *spawn* on final). M10.1.1's `HasMadeInitialContact` gate already handles this *if* the initial pattern check-in has set the flag. Verify the call ordering.
- **Closed-traffic vs other VFR pattern work.** "Request closed traffic" is for repeated touch-and-goes / stop-and-goes. "Request the option" is more flexible (touch-go, low approach, full stop, missed approach — pilot's choice). "Request stop-and-go" is single-attempt. Decide which intent is the default; the others can be future variants.
- **Initial-call fixed text vs scenario-supplied intent.** A scenario might preset specific intent ("Cessna 152 inbound for the option"). For MVP, hardcode "request closed traffic"; add a scenario field if real scenarios need it.
- **Per-leg state field placement.** Mirror `HoldingShortPhase._hasAnnouncedReady`: each pattern phase gets its own `_hasAnnounced` private bool, snapshot-serialized via the matching phase DTO. Verify each pattern phase has a DTO.
- **IFR pattern work.** Rare but exists (military, training). Defer leg announcements for IFR — only VFR fires them. Initial closed-traffic request also gated to VFR.
- **VfrFollowPhase.** Spacing-related — does it represent a pattern leg or a trailing position? Probably no pilot speech needed; verify when active.

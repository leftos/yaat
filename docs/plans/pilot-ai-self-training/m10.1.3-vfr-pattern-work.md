# M10.1.3 — VFR closed-traffic pilot speech

**Status:** shipped (2026-05-02)
**Size:** Small
**Maturity:** complete

## What shipped

Three new pieces of pilot speech for VFR closed-traffic pattern work at towered fields, all gated by `SoloTrainingMode && IsVfr` and routed through `AircraftState.PendingNotifications`:

- [x] **Initial closed-traffic request** on `PatternEntryPhase.OnStart` when `!HasMadeInitialContact`. Anchors distance/bearing on the runway threshold; includes altitude.
  Output: `"[N123AB] tower, november one two three alpha bravo, three miles south at one thousand five hundred, request closed traffic, with information Alpha."`

- [x] **Midfield-downwind reminder** at midfield-no-clearance. Mode-branched off the existing `_midfieldBroadcastIssued` block in `DownwindPhase` — solo+VFR voices the reminder as pilot speech; everything else (RPO mode, IFR pattern aircraft) keeps the existing `PendingWarnings` controller-facing nag.
  Output: `"[N123AB] november one two three alpha bravo, midfield downwind runway two eight right."`

- [x] **Short-final reminder** at distNm ≤ 1.0 with no landing clearance, for `_isPatternTraffic` aircraft. Mode-branched off the existing `_noClearanceWarningIssued` block in `FinalApproachPhase` — solo+VFR+pattern voices the reminder as pilot speech; everything else (RPO mode, IFR, non-pattern) keeps the existing warning.
  Output: `"[N123AB] november one two three alpha bravo, short final runway two eight right."`

Also:
- [x] M10.1.1 spawn-on-final check-in additionally gated by `!_isPatternTraffic` (defense in depth — pattern traffic uses `PatternEntryPhase` for initial contact, not `FinalApproachPhase`).
- [x] `PatternEntryPhase` snapshot serializes `_hasAnnouncedInitialCall` via new `PatternEntryPhaseDto.HasAnnouncedInitialCall` field (non-required for backward compat).

## Realism rule established

At towered fields, pilots in closed traffic do **not** self-announce every leg at phase entry. They speak twice:

1. **Initial contact** with intent (`"request closed traffic"`) — on first phase activation regardless.
2. **Reminder transmissions** at midfield-downwind or short-final **only if** tower hasn't cleared them yet. This is the pilot prompting tower to issue a clearance.

The same trigger drives both modes — `SoloTrainingMode` selects the output channel:
- `PendingNotifications` (pilot speech) in solo mode
- `PendingWarnings` (controller-facing nag, existing behavior) in RPO mode

This rule is recorded in the overarching plan README's "Decisions committed" section, plus a memory note (`project_towered_pilot_speech_no_clearance_driven`) so future M10.1.x work follows the same pattern.

## Decisions taken (during planning, 2026-05-02)

- **Initial call includes altitude** — matches `BuildAirborneCheckIn`'s VFR-inbound format for consistency.
- **MVP intent is "request closed traffic"** — option / stop-and-go / low-approach variants deferred until scenarios need them.
- **Mid-pattern-spawn deferred** — aircraft that spawn already on downwind get M10.1.2's airborne-spawn check-in (which says "inbound for landing" instead of "request closed traffic" — slightly inaccurate but acceptable for MVP).
- **No base announcement** — pilots don't normally call base at towered fields; controller either says "turn base" or just clears them.
- **`_hasAnnouncedShortFinal` not needed** — reused the existing `_noClearanceWarningIssued` flag in `FinalApproachPhase` rather than adding a new field. Same trigger (1 NM, uncleared), just mode-branched output channel. Saves a DTO field.
- **"Short final" kept over FAA-canonical "one mile final"** — aviation review (post-implementation) flagged that 7110.65 §3-10-1 prescribes "(distance) mile final" for controller-issued reports, but pilot transmissions don't have to mirror controller phraseology. "Short final" is the GA-pilot colloquialism heard universally on real LiveATC. Project principle: pilot-side speech ≠ 7110.65 controller form.
- **"Inbound, " dropped from initial call** — aviation review flagged it as redundant with "request closed traffic" (the request statement already conveys intent). Both forms are heard in real ops; the cleaner phrasing was adopted.

## Out of scope (deferred to future work)

- Upwind / crosswind / base leg announcements
- "Request the option" / "stop-and-go" / "low approach" intents
- Scenario-configurable intent enum
- Dynamic ATIS letter (replace hardcoded "Alpha")
- Special-casing M10.1.2's airborne check-in for mid-pattern-spawn aircraft (would say "request closed traffic" if `TrafficDirection != null`)
- IFR pattern traffic speech (rare; gated out by `IsVfr`)
- VfrFollowPhase speech (covered by `PatternEntryPhase` on join — VfrFollow self-replaces with `PatternEntryPhase + circuit` once it joins)

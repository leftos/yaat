# M11.5 — Frequency-stuck pilot (failed handoff check-in)

**Status:** Stub — not yet planned. Flesh out before implementing.

## Why

When `CT` (contact next controller) is issued the simulated pilot acknowledges and the workflow ends cleanly. In real ops, occasionally a pilot mis-sets the radio and never checks on with the new sector — the receiving controller calls the previous sector to find them, and the previous sector calls the aircraft back on the old frequency to re-issue the contact. This is a rare but distinctive workflow.

## Sketch

- Rare event (~2% of `CT` instructions) flags the aircraft as "switched but never checked on with next sector." The aircraft continues on its filed/cleared profile but is silent on the new frequency.
- After ~60-90 seconds of silence, the receiving sector (simulated in solo training as a pop-up coordination event) calls the previous sector: "Approach, Center — any joy on N12345? Haven't heard from them."
- Player options:
  - Recall the aircraft to the previous frequency (new command: `RECALL [callsign]` or use existing `CT` re-issue).
  - The aircraft acknowledges, then the player re-issues `CT` and the new check-in happens (no second failure).
- In solo mode without a real receiving sector, this is simulated as a one-line terminal event from "next sector" prompting the player.

## Open questions

- How is this distinguished from the player simply forgetting to `DROP` the aircraft? Probably: failure-mode events emit a clear "[NCT] any joy on N12345?" terminal line, whereas player oversight is silent.
- Does the aircraft eventually self-resolve (pilot realizes, calls back) if the controller never recalls? Realistic time scale would be 2-3 minutes — long enough to feel like real consequences.
- Does this interact with handoff scoring (M12.3)? Probably yes — losing the aircraft entirely should ding.
- Is this only an IFR scenario? (VFR handoffs are less common in the workflow we model.)

## Out of scope (probably)

- Modelling the *next* sector actually working the aircraft. Beyond the one prompt, we don't simulate the receiving sector.
- Full lost-comms (separate milestone M11.6 covers the NORDO case).

## Touches

- `src/Yaat.Sim/Commands/ContactCommandHandler.cs` — perturbation hook on `CT`.
- `src/Yaat.Sim/Aircraft/AircraftState.cs` — "check-in failed" timer.
- Some "external coordination event" surface — probably a new lightweight `CoordinationPrompt` channel that delivers terminal events without involving the full SignalR coordination protocol.

## Pick-up notes

Edge-case feature — confirm value with the user before building. If we do build it, the simplest version is: emit the "any joy?" terminal line after the timer fires, let the player respond with a re-issued `CT`, and call it done. Skip the full self-resolution path.

# Plan: vTDLS Operational Configurations + dropdown placeholder

## Context

vNAS shipped Ops Configs on 2026-06-22 (announced by ZBW DATM Nathan Rankin). A Facility
Engineer can now attach a **different SID list — and different per-transition defaults — to
each operational configuration**, so an east/west runway split issues different departure
frequencies, maintain altitudes, or local info. The controller picks the active config from
an **Ops Config menu in the footer** and clicks **Save**.

YAAT parses none of this today: `rg OpConfig` returns zero hits across both repos.

### Why this is urgent, not cosmetic

When `dclOpConfigsEnabled` is true the facility-level `sids` array is **empty** — every SID
moves into the ops configs. Live data as of 2026-07-22:

| ARTCC | Facility | `dclOpConfigsEnabled` | ops configs | facility-level `sids` |
|---|---|---|---|---|
| ZOA | **SFO** | true | 7 (`2801`, `2828RT`, `2828SO`, `0101`, `1910`, `1919`, `1010`) | **0** |
| ZOA | **OAK** | true | 3 (`OAKW`, `OAKE`, `SFOE`) | **0** |
| ZBW | **BOS** | true | 2 (`Logan Sid`, `FDT`) | **0** |
| ZOA | SJC / SMF / RNO | false | 0 | 7 / 5 / 6 |
| ZDC | IAD / DCA / BWI / ADW / RDU | false | 0 | 10 / 10 / 7 / 5 / 10 |

So **YAAT's vTDLS currently shows an empty SID dropdown at SFO, OAK and BOS** — the three
busiest fields in our own scenarios and docs. (`mandatorySid` is false at all three, so Send
isn't hard-blocked the way #306 was, but no SID can be issued at all.)

The defaults genuinely differ per config — this is the feature's payload, not a rename:

```
SFO  TRUKN2.GRTFL  [2801]   … localInfo "EXPECT RWY 1R"
     TRUKN2.GRTFL  [2828RT] … localInfo "EXPECT RWY 28L"
     TRUKN2.GRTFL  [0101]   … localInfo "EXPECT RWY 1L"
BOS  BLZZR6.- - - - [Logan Sid] … contactInfo "CTC $FREQ TO PUSH"
     BLZZR6.- - - - [FDT]       … contactInfo "CTC 121.65 TO PUSH"
```

### Confirmed wire model

`OpConfig` is identical in the vNAS `data` lib (`data/Facilities/Tdls/OpConfig.cs`, up to
date at 2026-06-20) and in freshly decompiled **CRC 2.17.4** — the decompile added a new
`Vatsim.Nas.Data.Facilities.Tdls/` namespace and two fields on `TdlsConfiguration`:

```csharp
public class OpConfig {
    public required string Id, Name;
    public required List<TdlsSid> Sids;      // full SID list incl. per-transition defaults
    public string? DefaultSidId, DefaultTransitionId;
}
// TdlsConfiguration gained:  bool DclOpConfigsEnabled;  List<OpConfig> OpConfigs;
```

CRC carries the model but has **no** selection logic — it never renders TDLS. The active
config is purely a vTDLS concern.

On the wire the live object carries only `{id, name, sids}`; `defaultSidId` /
`defaultTransitionId` are omitted when null (they are null for every config sampled).

### Footgun: SID ids are not stable across configs

For the same SID *name*, the id differs per config at OAK (12/12) and BOS (9/9), while SFO
happens to reuse one id across all 7. `ToClearanceDto` sends `SelectedSid.Id`, so an id is
only meaningful **relative to the config that was active when it was chosen**.

### Decisions taken

- **Active config is server room state** — per facility in `TdlsState`, broadcast, and
  snapshotted, following the bookmarks (#288) / TIMER precedent. PDC content depends on it,
  so instructor, trainee and every vTDLS window must agree, and a replay must rebuild the
  clearance that was actually sent.
- **Changing the active config closes any open editor.** It is a deliberate, Save-gated act;
  reopening yields clean defaults. Avoids a half-migrated clearance whose `SelectedSid.Id`
  no longer exists in the new config.
- **`- - - -` is placeholder-only** (`ComboBox.PlaceholderText`), matching upstream's "no
  option is selected, or none are available". No synthetic selectable row: across ZBW/ZDC/ZOA
  **no** value list contains an empty entry, while 9 lists already carry an FE-authored
  `- - - -` row — a synthetic one would duplicate those.
- **Refresh every vNAS doc mirror first**, before any TDLS code.

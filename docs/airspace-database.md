# Airspace Database & Boundary-Crossing Detection

> Read this before touching anything under `src/Yaat.Sim/Data/Airspace/`, `AirspaceBoundaryHoldPhase`,
> `PilotProactive.TickAirspaceBoundaryRespect`, or the Class B/C separation-minima path in
> `SoloTrainingEvaluator`. This is the point-in-polygon containment engine plus its two consumers.
> The two tick-loop call sites live in **both repos** — a change here usually has to be mirrored.

## What it is (and is not)

`AirspaceDatabase` is a lateral point-in-ring + vertical altitude-band containment engine over a fixed set
of FAA Class B and Class C airspace volumes, loaded from a checked-in GeoJSON fixture. It answers three
questions for solo-training pilot AI and scoring:

1. *Is this aircraft currently inside a Class B/C volume?* — `FindContaining(position, altitudeFtMsl)`.
2. *Will this aircraft's projected track cross into a Class B/C volume within N seconds?* — `FindFirstProjectedEntry(...)`.
3. *Which separation minima apply to this pair right now?* — used indirectly by `SoloTrainingEvaluator`.

It is **not** the ARTCC facility / position / TCP / beacon-code config (that is `ArtccConfigService` /
`NavigationDatabase`), and it is **not** general airspace classification. Only **Class B and Class C** are
parsed — `ParseFeature` returns `null` for every other `CLASS` value, so Class D/E and airport features in
a source GeoJSON are silently dropped (see Footguns). All file references below are
`src/Yaat.Sim/Data/Airspace/` unless stated otherwise.

## Data model

Five small types in `Data/Airspace/`:

| Type | File | Shape |
|---|---|---|
| `AirspaceVolume` | `AirspaceVolume.cs` | `Id`, `Ident`, `IcaoId`, `Name`, `Class`, `LowerFtMsl`, `UpperFtMsl`, `Rings` (all `required init`). Owns the containment geometry. |
| `AirspacePoint` | `AirspacePoint.cs` | `readonly record struct (double Lat, double Lon)` — note **Lat first**. `ToLatLon()` / `FromLatLon()`. |
| `AirspaceClass` | `AirspaceClass.cs` | `enum { Bravo, Charlie }` — **two values only**. |
| `AirspaceBoundaryCrossing` | `AirspaceBoundaryCrossing.cs` | Result of a projected-entry query: `Volume`, `Intersection`, `DistanceNm`, `LookaheadSeconds`, `TimeToEntrySeconds`, `EntryAltitudeFtMsl`. |
| `AirspaceDatabase` | `AirspaceDatabase.cs` | `IReadOnlyList<AirspaceVolume> Volumes` + the lazy `Default` singleton + all loading/parsing/query methods. |

`Rings` is `IReadOnlyList<IReadOnlyList<AirspacePoint>>` — one volume can carry several rings (multiple
disjoint polygons, e.g. the surface area plus the shelf rings of a Class C). All rings are treated as
**positive** (see the doughnut footgun).

## Fixture loading & discovery

`AirspaceDatabase.Default` is a process-wide `Lazy<AirspaceDatabase>` (`AirspaceDatabase.cs:12`) populated by
`LoadDefault()` on first access. `LoadDefault` runs a **3-tier file search** for `*.geojson` and `*.geojson.br`:

1. `AppContext.BaseDirectory/Data/Airspace` (the content-copied fixture next to the built assembly).
2. `AppContext.BaseDirectory` itself (bare build output).
3. `FindFixturesFromWorkingTree()` — walk **up** from `Directory.GetCurrentDirectory()` looking for
   `src/Yaat.Sim/Data/Airspace`, for the source-tree / wrong-cwd case.

If all three tiers come up empty it logs a warning (`"No FAA airspace fixtures found under {Path}"`) and
returns an **empty database** — every downstream airspace query then silently no-ops (see Footguns).

`FromGeoJsonFiles` reads the matched files in case-insensitive sorted order and **dedups by volume `Id`**
across files via `volumesById.TryAdd` (first file wins on a collision). `ReadGeoJsonText` auto-detects
compression by extension: a `.br` path is decompressed through a `BrotliStream`; anything else is read as
plain UTF-8. The shipped fixture is Brotli.

### The fixture and its refresh script

The repo ships one fixture: `Data/Airspace/faa-training-primary-class-bc.geojson.br` (~6.1 MB Brotli). It is
regenerated by `tools/refresh-faa-airspace.ps1`, which:

- Resolves the primary airports of the YAAT training scenarios across 20 default ARTCCs (`ZAB … ZTL`) via
  the vNAS training API (`scenario-summaries/by-artcc/{artcc}` → each scenario's `primaryAirportId`),
  normalizing K-/P-prefixed idents down to the bare 3-letter form.
- Queries the FAA ArcGIS `Class_Airspace` FeatureServer for `CLASS in ('B','C')` and `IDENT in (...)` for
  those airports (in chunks of 30 idents), keyed by `OBJECTID`.
- Writes the merged `FeatureCollection` to the `-Output` path, Brotli-compressing when the path ends in `.br`.

The script defaults to Classes `B` and `C` only — consistent with `ParseFeature` dropping everything else,
the parser would discard any other class even if the fixture carried it.

## GeoJSON parsing

`FromGeoJson` parses the top-level `FeatureCollection`, calling `ParseFeature` per feature. `ParseFeature`:

- Maps `properties.CLASS`: `"B" → Bravo`, `"C" → Charlie`, anything else → `null` (feature dropped).
- Builds `Id` as `FAA-AIS-{OBJECTID}` when `OBJECTID > 0`, else a composite `"{Ident}-{Name}-{lower}-{upper}"`.
- Reads `IDENT`, `ICAO_ID`, `NAME` (falling back to `IDENT`).
- Resolves the altitude band via `ResolveAltitudeFt(props, "LOWER", 0)` and `ResolveAltitudeFt(props, "UPPER", int.MaxValue)`.

`ResolveAltitudeFt` (`AirspaceDatabase.cs:318`) reads the `{PREFIX}_CODE` / `{PREFIX}_VAL` property pair:
`_CODE == "SFC"` → `0`; a missing value or a sentinel `<= -9990` → the supplied default
(`0` for LOWER, `int.MaxValue` for UPPER); otherwise `round(_VAL)`.

`ParseRings` flattens geometry: `Polygon` coordinates go straight to `ParsePolygon`; `MultiPolygon`
coordinates are iterated polygon-by-polygon into the same flat ring list. `ParsePolygon` reads each
coordinate pair as **`[lon, lat]`** — it stores `new AirspacePoint(pair[1] /* Lat */, pair[0] /* Lon */)`.
Rings with fewer than 4 points are discarded. A feature whose rings all drop out yields no volume.

## Containment geometry

`AirspaceVolume.Contains(position, altitudeFtMsl)` is `ContainsAltitude && ContainsLateral`:

- `ContainsAltitude(alt)` = `alt >= LowerFtMsl && alt <= UpperFtMsl` (inclusive both ends).
- `ContainsLateral(position)` = `PointInRing` true for **any** ring (rings OR-combine).

`PointInRing` (`AirspaceVolume.cs`) is a standard even-odd ray-cast in `(Lat, Lon)` space, returning
`false` for any ring with fewer than 3 points. `FindLateralIntersections(from, to)` walks each ring's edges
and calls `GeoMath.SegmentsIntersect(from, to, a, b, excludeEndpoints: false)`, yielding each hit point —
this is the lateral-crossing input to projected-entry detection.

**Bounding-box pre-filter (hot path).** Each `AirspaceVolume` caches its rings' axis-aligned `(Lat, Lon)`
bounding box (computed once when `Rings` is assigned). `ContainsLateral` and `FindLateralIntersections`
reject in O(1) when the point / segment lies entirely outside that box *before* walking any ring vertices.
The box is a superset of the polygon in the same planar space as the exact test, so results are identical —
it only skips provably-outside work. This matters because `Default` is the **national** Class B/C set (≈500
volumes / ≈665k vertices) and the solo-training per-tick paths (`FindFirstProjectedEntry` per VFR aircraft,
`FindContaining` per aircraft pair) sweep every volume; without the bbox reject each sweep is a six-figure
point-in-polygon walk. Never iterate `Volumes` with a raw `PointInRing`/segment test in a per-tick loop —
go through `Contains` / `FindLateralIntersections` so the pre-filter applies.

## Projected-entry detection — `FindFirstProjectedEntry`

`FindFirstProjectedEntry(aircraft, lookaheadSeconds)` (`AirspaceDatabase.cs:26`) returns the **earliest**
boundary crossing along the aircraft's projected straight-line track, or `null`. It projects the endpoint
position (`ProjectPosition`) and altitude (`ProjectAltitude`) for the full lookahead, then for each volume
considers **three candidate modes** and keeps the one with the smallest `TimeToEntrySeconds`:

1. **Already-inside-at-projected-point** — if `volume.Contains(to, projectedAltitude)`, consider `to` at the
   full lookahead time.
2. **Lateral ring intersection** — for each `FindLateralIntersections(from, to)` hit, time-to-entry is
   proportional to along-track distance (`clamp(distanceNm / lookaheadNm, 0, 1) * lookaheadSeconds`); the
   altitude *at* that time is re-projected and the candidate is taken only if `ContainsAltitude` holds there.
3. **Vertical floor/ceiling crossing** — `ConsiderVerticalEntry` for both `LowerFtMsl` and `UpperFtMsl`:
   requires `|VerticalSpeed| >= 1` fpm, computes `timeToEntry = |Δalt / vs| * 60` (min → s), rejects if outside
   `[0, lookaheadSeconds]`, projects the lateral position at that time, and requires it to be laterally inside.

**Skip-if-already-inside guard:** before any of the three modes, the loop `continue`s on any volume the
aircraft is *already* in (`volume.Contains(from, aircraft.Altitude)`). So this method only ever reports the
**first entry** of a not-yet-entered volume — never re-entry, never exit. Use `FindContaining` for plain
"am I inside" questions (see Footguns).

### Trajectory math — `ProjectPosition` / `ProjectAltitude` / `ResolveAltitudeGoal`

`ProjectPosition` advances along `TrueTrack` by `max(GroundSpeed, 0) * lookahead / 3600` NM via
`GeoMath.ProjectPoint`. `ProjectAltitude` advances by `VerticalSpeed * lookahead / 60`, then **clamps toward
the altitude goal** from `ResolveAltitudeGoal` so a climbing/descending aircraft does not overshoot its
target/floor/ceiling. `ResolveAltitudeGoal` (private, `AirspaceDatabase.cs:152`) folds
`Targets.TargetAltitude` together with `AltitudeFloor`/`AltitudeCeiling`.

This is one of two `ResolveAltitudeGoal` implementations — the other is inside `FlightPhysics.UpdateAltitude`
(with `AltitudeSnapFt` hysteresis). They intentionally differ (projection wants the raw trajectory, physics
wants snap hysteresis) but must stay in sync on floor/ceiling resolution. The physics-side integration math
(airspeed frames, altitude snap, performance constants) is owned by [flight-physics.md](flight-physics.md);
this doc does not re-derive it. This is also flagged in
[solo-training-pilot-speech.md](solo-training-pilot-speech.md) (Pitfalls).

## Consumer 1: VFR boundary respect (the hold)

`PilotProactive.TickAirspaceBoundaryRespect(aircraft, scenario, airspace, airportLookup)`
(`src/Yaat.Sim/Pilot/PilotProactive.cs:175`) models AIM §3-2-1.4 pilot responsibility: a VFR student must
meet Class B/C entry requirements before crossing the boundary. It early-returns unless **all** of:

- `scenario.SoloTrainingMode` is on,
- the aircraft is **airborne and VFR** (`!IsOnGround && FlightPlan.IsVfr`),
- there is **no active phase** (`aircraft.Phases is { IsComplete: false }` returns).

Then it calls `airspace.FindFirstProjectedEntry(aircraft, lookaheadSeconds: 60)`. If a crossing is found and
`EntryGateSatisfied` is **false** for that volume's class, it installs a fresh single-phase `PhaseList`
holding an `AirspaceBoundaryHoldPhase` configured from the volume (class, ident/icao, name, reference
position via `airportLookup`, right-orbit, lower/upper band).

`AirspaceBoundaryHoldPhase` (`src/Yaat.Sim/Phases/AirspaceBoundaryHoldPhase.cs`) overrides
`ManagesSpeed => true` and on `OnStart` snapshots the pre-hold route/heading/turn/speed, clears the nav
route, and orbits: `SetHoldingTargets` caps speed to `AircraftPerformance.HoldingSpeed(...)` and turns the
aircraft 180° in the orbit direction. The hold is **silent** — it does not broadcast a pilot self-report
(Issue #154). `OnTick` completes (returns `true`) when either `GateSatisfied` is met or
`HeldVolumeCanStillBeEntered` becomes false (the held shelf is no longer vertically relevant, using a 60 s
`AirspaceDatabase.ProjectAltitude` band check). On completion `OnEnd` **restores the original route/heading
only if the controller has not since assigned an explicit magnetic heading and has not added a new
navigation route** (a non-empty `NavigationRoute` at `OnEnd` means a `DCT`/direct command was issued during
the hold, which must survive the release), and restores the original speed only if there is no explicit
speed command. `CanAcceptCommand` returns `Allowed` for every command (the hold never rejects controller
input). See [phases.md](phases.md) for the phase lifecycle, `ManagesSpeed`
contagion, and the snapshot contract; see
[plans/pilot-ai-self-training/m10.1.5-vfr-airspace-respect.md](plans/pilot-ai-self-training/m10.1.5-vfr-airspace-respect.md)
for milestone history.

### Entry gates

`EntryGateSatisfied` (in `PilotProactive`) and the identical `GateSatisfied` (in the hold phase) read
boolean gate flags on `AircraftState`:

| Class | Gate condition | Backing field(s) |
|---|---|---|
| `Bravo` | explicit Class B clearance issued | `IsClearedIntoBravo` |
| `Charlie` | two-way comms established | `HasMadeInitialContact && HasControllerAcknowledgedInitialContact` |
| _other_ | `true` (no gate) | — |

These flags live on `AircraftState` (`IsClearedIntoBravo` at `AircraftState.cs:219`,
`HasControllerAcknowledgedInitialContact` at `:169`, `HasMadeInitialContact` at `:162`) and are
snapshot-serialized so replays preserve gate state.

`HasControllerAcknowledgedInitialContact` is set whenever the student issues a successful command
(`RoomEngine.SendCommandAsync`, mirrored in `SimulationEngine.SendCommand`). `HasMadeInitialContact` is
normally set by the AI pilot's proactive check-in, but that check-in is suppressed when the pilot cannot
initiate contact with the student (`PilotInitialContactEligibility.CanInitiateWithStudent` — track owned by
another position with no handoff inbound, e.g. an S2 tower scenario whose arrivals stay with approach). In
that case the AI pilot would never speak first, so a student command also marks `HasMadeInitialContact`:
both call sites route through `PilotInitialContactEligibility.RegisterControllerContact(aircraft, scenario)`,
which sets the pilot side only when `CanInitiateWithStudent` is false. This is what lets the Charlie gate
clear when a tower student directs an approach-owned VFR arrival into the Class C.

## Consumer 2: separation-minima selection

`SoloTrainingEvaluator.ResolveRequirement(a, b, airspace, lookaheadSeconds)`
(`src/Yaat.Sim/Training/SoloTrainingEvaluator.cs:693`) picks which separation standard a pair must meet. The
airspace database drives the VFR-involved branches:

- **Both IFR** → IFR radar separation (3 NM / 1,000 ft), no airspace lookup.
- **A volume contains either aircraft (projected)** via `FindApplicableAirspaceClasses` →
  `FindContainingProjectedAirspace` → `airspace.FindContaining(projectedPosition, projectedAltitude)`:
  - **Class B present** → `IsLargeOrTurbojet(a) || IsLargeOrTurbojet(b)` selects between **1.5 NM / 500 ft**
    (large >19,000 lb or turbojet, §7-9-4) and **target-resolution / 500 ft** (light, §7-9-4).
  - **Class C present, or the synthetic outer-area applies, and the pair is mixed IFR/VFR** →
    Class C target-resolution / 500 ft (§7-8-2/3, AIM §3-2-4).

Note that `SoloTrainingEvaluator` has its **own private** `ProjectPosition`/`ProjectAltitude`
(`SoloTrainingEvaluator.cs:1391`+) — they are *not* the `AirspaceDatabase` ones, and `ProjectAltitude` here
does no goal clamping. See [solo-training-evaluation.md](solo-training-evaluation.md) (the Separation criteria
section) for how these requirements feed the scorer and the per-tick `Evaluate` entry point.

### The synthetic Class C outer area

The Class C "outer area" (the 20 NM ring used to extend §7-8 separation responsibility) is **not a polygon in
the GeoJSON**. `IsInClassCOuterArea` (`SoloTrainingEvaluator.cs:756`) groups the Charlie volumes by airport id
(`IcaoId` else `Ident`), looks the airport fix up in `NavigationDatabase` trying both the raw id and a
`"K"`-prefixed variant, takes the group's max `UpperFtMsl` as the ceiling, and returns true when the projected
position is within a **hardcoded 20.0 NM radius** and at-or-below that ceiling. If the airport fix is missing
from navdata, the outer-area check silently returns `false`.

## Tick-loop wiring (cross-repo)

`TickAirspaceBoundaryRespect` is called once per aircraft in **PostPhysics** in *both* repos:

- yaat: `SimulationEngine.TickPostPhysics` (`src/Yaat.Sim/Simulation/SimulationEngine.cs:542`).
- yaat-server: `TickProcessor` (`../yaat-server/src/Yaat.Server/Simulation/TickProcessor.cs:287`).

Both pass `AirspaceDatabase.Default`. The separation path is wired separately in yaat-server at
`TickProcessor.cs:309` (`SoloTrainingEvaluator.Evaluate(..., AirspaceDatabase.Default, ...)`). See
[tick-loop.md](tick-loop.md) for where PostPhysics sits relative to physics sub-ticks. Adding or removing a
proactive airspace tick must be mirrored across both call sites or solo behavior diverges between the local
sim and server rooms.

## Footguns & gotchas

- **Only Class B and C are parsed.** `ParseFeature` returns `null` for any other `CLASS`, so Class D/E and
  airport features in a source GeoJSON are silently dropped. The boundary-hold and separation logic never
  fire for Class D even though the gate switches carry a permissive `_ => true` default.
- **The entry-gate switch is duplicated in (effectively) three places:** `AirspaceBoundaryHoldPhase.GateSatisfied`,
  `PilotProactive.EntryGateSatisfied`, and the implicit class checks in `SoloTrainingEvaluator.ResolveRequirement`.
  Add a class or change a gate rule and you must touch all of them — the compiler will not catch a missed one
  because the phase/proactive switches fall through to `_ => true`.
- **Two `ResolveAltitudeGoal` implementations.** `AirspaceDatabase.ProjectAltitude` (no snap hysteresis,
  projection wants raw trajectory) vs `FlightPhysics.UpdateAltitude` (with `AltitudeSnapFt`). They must stay in
  sync on floor/ceiling resolution. Also flagged in [solo-training-pilot-speech.md](solo-training-pilot-speech.md).
- **The Class C outer area is synthetic.** It is a hardcoded 20 NM radius around the navdata airport fix, not a
  polygon. If the airport fix is absent from `NavigationDatabase` (raw or `K`-prefixed), the check returns
  `false` and the extended separation responsibility silently does not apply.
- **Empty-DB silent no-op.** When fixture discovery finds nothing, `LoadDefault` logs a warning and returns
  `new AirspaceDatabase([])`. Every query then returns nothing and all airspace behavior disappears with no
  exception — a wrong cwd or a missing content-copy can disable the whole subsystem invisibly. The
  `FaaTrainingPrimaryFixture_LoadsSfoBravoAndOakCharlie` test (`tests/Yaat.Sim.Tests/AirspaceRespectTests.cs`)
  guards that the fixture actually loads.
- **`AirspaceDatabase.Default` is a process-wide `Lazy` singleton populated from disk** — like other Yaat.Sim
  singletons it can race in parallel xUnit if a test reads it mid-population (see the static-singleton-races
  note in `CLAUDE.md`).
- **GeoJSON coordinate order is `[lon, lat]`.** `ParsePolygon` reads `pair[1]` as Lat and `pair[0]` as Lon, and
  `AirspacePoint` stores `(Lat, Lon)`. Swap them and containment flips silently.
- **`FindFirstProjectedEntry` skips any volume the aircraft is already inside.** It reports only the first
  not-yet-entered boundary, never re-entry or exit. Do not repurpose it for "am I inside" — use `FindContaining`.
- **`ParseRings` treats every ring as a positive ring.** `ContainsLateral` OR-combines all rings, so a true
  polygon hole (a doughnut) would be treated as solid. Fine for the current SFO/OAK fixture, but a correctness
  trap if a doughnut-shaped volume is ever added.
- **The tick wiring is a cross-repo pair.** `SimulationEngine.TickPostPhysics` and yaat-server's
  `TickProcessor` both call into this subsystem with `AirspaceDatabase.Default`; keep them mirrored.

## Extending

- **Add Class D** — add `Delta` to the `AirspaceClass` enum, map `"D"` in `ParseFeature`, add the gate rule in
  all three switch sites (hold phase, proactive, evaluator), and re-run `tools/refresh-faa-airspace.ps1` with
  `-Classes B,C,D` so the fixture carries the volumes.
- **Refresh the fixture** — `pwsh tools/refresh-faa-airspace.ps1` (optionally `-Artccs`, `-Idents`, `-Classes`,
  `-Output`). It rediscovers training primary airports and re-queries the FAA ArcGIS service.
- **Add a query** — extend `AirspaceDatabase` with a method over `Volumes`; keep the `[lon, lat]` / `(Lat, Lon)`
  convention and remember the skip-if-already-inside semantics differ from plain containment.
